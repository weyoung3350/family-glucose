# 家有糖人 开发任务总览

> 总任务数：21（后端 10 + 前端 10 + 集成 1）
> 编写时间：2026-04-29
> 任务粒度：每个任务在 1-3 小时可被一名开发者（或 codex）完成。
> 验收方：Claude（基于 PRD + 概要设计 + 任务文档的验收清单）。

## 阅读顺序

每个任务交给 codex 之前，请按以下顺序提供上下文：

1. `docs/简化版血糖记录小程序-需求规格说明书.md`（PRD）
2. `docs/概要设计.md`
3. 该任务文档本身（`docs/tasks/Bxx-xxx.md` 或 `Fxx-xxx.md`）

任务文档已自包含足够信息，但前两份是最终事实来源。如遇任务文档与 PRD/概要设计冲突，以 **PRD > 概要设计 > 任务文档** 的优先级为准。

## 任务文档结构约定

每个任务文档包含以下小节，便于 codex 一次读完即可动手：

1. **任务编号 / 标题**
2. **目标**（一句话）
3. **前置依赖**（任务编号列表）
4. **上下文**（指向 PRD / 概要设计相关章节）
5. **输入**（已有文件 / 配置 / 接口契约）
6. **输出**（要新增 / 修改的文件清单 + 关键文件示例骨架）
7. **详细需求**（功能、字段、错误处理）
8. **验收标准**（可勾选 checklist，Claude 用此清单逐项验收）
9. **不在范围内**（out of scope，避免代码越界）

## 验收流程

每个任务由 codex 实现完成后：

1. 用户把 codex 的 PR 或代码提交到工作目录
2. 用户告诉 Claude "验收 BXX" 或 "验收 FXX"
3. Claude 按任务文档的"验收标准"清单逐项核对，必要时执行测试
4. 输出验收报告：`tasks/_acceptance/BXX-验收报告.md`，记录通过 / 不通过的项与建议
5. 若不通过，把建议带回 codex 修订
6. 所有 BXX/FXX 通过后才进入 T01

## 项目变更

任何对**已锁定决策**（见 `项目对话记录.md` 表格）的修改，必须先更新 PRD/概要设计/对话记录，再标记影响范围内的任务为"待重做"。
