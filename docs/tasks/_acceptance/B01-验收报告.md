## 观察到的小问题（不阻塞，仅提醒）

### O1：`.env.example` 与 Settings 默认值对 APP_ENV 不一致

- `app/config.py` 第 13 行：`APP_ENV: str = "development"`
- `.env.example` 第 2 行：`APP_ENV=production`

后果：开发者按 README 走 `cp .env.example .env`，再 `uvicorn ...` 启动后看到 `env: production`。这与"开发者本地默认应该是 development"的直觉不符。

建议：`.env.example` 改为 `APP_ENV=development`。production 由部署脚本（B10）覆盖。**不要求 codex 重做 B01**，可在 B10 写部署 README 时一并提示，或由后续微调任务捎带处理。

### O2：HTTPException 的 str detail 形式 message 等于 code

handler 当 detail 是 `"ERR_TEST"` 这类字符串时，message 也直接用了 `"ERR_TEST"`，对前端展示不友好。

建议：**后续 B 任务（B03-B09）抛业务错误统一用 dict detail**：

```python
raise HTTPException(400, {"code": "ERR_INVITE_CODE_INVALID", "message": "邀请码不存在"})
```

handler 已经正确支持这种形式（实测通过），无需改动。这条作为 **B03+ 的开发约定**记录。

### O3：依赖列表中 `httpx` 出现在主依赖也出现在 dev extras

`pyproject.toml` 第 14 行（主）和第 24 行（dev）都列了 `httpx`。无害（pip 会去重），但稍冗余。

建议：dev extras 移除 `httpx`，因为它已经是主依赖。**不影响功能**，可在 B03 实现 `services/wechat.py` 时顺手清理。

## 后续 B 任务的开发约定（基于本次验收沉淀）
