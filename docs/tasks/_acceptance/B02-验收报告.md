## 观察到的小问题

### O1：families.creator_id 改为 NULLABLE（合理妥协，但需后续保证）

任务文档 §5.1 期望 `creator_id NOT NULL`，codex 改成 `Optional[int]` 即 NULL 允许。

**判断**：可以接受。families 与 users 之间是循环外键，业务流程是"创建 user → 创建 family → 关联回去"，先 INSERT family 时 creator_id 必然为 NULL，紧随其后的 UPDATE 才能填入。这是 ORM 在 SQLite 上的常规处理。

**约定（写入 B04 开发约定）**：
- B04 实现 `POST /families` 必须在事务内确保 creator_id 在 commit 之前被填入
- 代码层每次读 family 时假定 creator_id 非空（家庭一定有管理员）
- 添加单元测试：DB 层不约束，但应用层强制非空

### O2：glucose_records 的 measured_at 出现两个索引

`measured_at` 既有单列索引 `ix_glucose_records_measured_at`，又被复合索引 `ix_glucose_records_family_measured_at` 覆盖。后者已能服务 measured_at-only 查询（前缀匹配规则：单字段时无效，但 family 已是常量）。

实际上 measured_at-only 查询不会发生（所有查询都加 `family_id` 限制），所以单列索引冗余。**不阻塞**。

**建议**：B05 实现完成后回头看是否真的有不带 family_id 的 measured_at 查询，如无则删除单列索引。

### O3：复合索引未指定 DESC

任务文档要求 `INDEX(family_id, measured_at DESC)`。实际是 ASC。SQLite 索引可双向遍历，对 `ORDER BY measured_at DESC` 查询的性能等价，**不影响功能**。家庭量级（一年 ≤ 11000 条）也不会产生可观察的差距。

### O4：VARCHAR vs TEXT

期望 TEXT，实际 VARCHAR。**这是 SQLModel/SQLAlchemy 的默认 str 映射，SQLite 内部把 VARCHAR 当 TEXT 存储，等价**。无需改。

## B03 开发约定（基于本次验收沉淀）

1. 创建 family 必须在同一事务里把 creator_id 填上
2. 业务读取 family 时假定 creator_id 非空
3. CheckConstraint 在 ORM 层若想做更友好的错误提示，应该在 schema/router 层先校验，DB 层兜底
4. utcnow 的使用：所有 datetime 入库为 UTC，前端展示按本地时区转（前端已知）
