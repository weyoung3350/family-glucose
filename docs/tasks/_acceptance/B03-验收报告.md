## B01-O2 约定遵循情况

✅ **完美遵循**。`deps.py:raise_auth_error` 强制 dict 形式：

```python
def raise_auth_error(code: str, message: str, status_code: int = 401) -> None:
    raise HTTPException(
        status_code=status_code,
        detail={"code": code, "message": message},
    )
```

`routers/auth.py` 中所有 HTTPException 抛错也都用 dict detail，message 是中文人话（"登录 code 无效"、"微信登录服务异常"、"请求体格式无效"）。

## 观察到的小问题

### O1：概要设计 §9.3 解释文字与字符集不一致（**非代码问题，文档需修**）

`docs/概要设计.md` 第 9.3 节原文：
> 字符集 `ABCDEFGHJKLMNPQRSTUVWXYZ23456789`（去 **0/O/1/L/I**），长度 6

字符集实际只剔除了 `0/O/1/I`，**保留了 L**。括号注释里多写了一个 L。codex 按字符集字面执行，是正确的。

**建议**：把概要设计 §9.3 括号注释改为"去 0/O/1/I"（不动代码）。L 与已剔除的 1/I 大写形态视觉区分度足够，留着无害。这条变更我会在下次需要修文档时一并处理。

### O2：JWT payload 多带了 family_id 与 role（任务文档未要求）

任务文档 §security.py 写的是 `{"sub": user_id, "openid": openid_last4, "exp": ...}`，codex 多加了 `family_id` 和 `role`。

**判断**：可保留。get_current_user 中间件每次都从 DB 重新查 user，**不依赖 token 里的 family_id/role**，所以即使 token 中字段过期也无副作用。多带这两个字段对前端调试有点用（可以 jwt.decode 一眼看出当前角色），无安全/正确性影响。

**建议**：保留现状，但在概要设计 §7.1 备注中加一句"JWT 可携带 family_id/role 作为前端调试信息，但服务端始终以 DB 为准"。

### O3：parse_login_request 用了自定义解析器（任务文档未要求）

codex 没用 FastAPI 默认的 `body: LoginRequest`，而是写了一个 `parse_login_request` 依赖手动解析 JSON，并把 schema 错误统一转 `{"code":"ERR_HTTP","message":"请求体格式无效"}`。

**判断**：合理改动。FastAPI 默认 422 + 详细字段错误清单对小程序前端不友好，这种统一封装符合 §12 错误格式。**保留**。
