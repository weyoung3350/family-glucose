## 0. 用户问题答复

> **问：我创建的家庭，里面的血糖记录会不会被没加入家庭的用户看到？**

**答：不会。** 核心数据隔离机制是正确的——所有读取血糖记录的接口都强制查询 `family_id = current_user.family_id`，只要这个 `family_id` 不被冒用，外人就看不到你的数据。

但**这个保证依赖几个前置条件**，目前都有不同程度的风险敞口（详见第 2 节）：

1. JWT_SECRET 配置正确（缺省 `""` 是灾难）
2. 邀请码不被猜中或泄漏（无 rate limit + 永久有效）
3. CSV 导出的 query token 不被截获（URL 中带完整 30 天 JWT）
4. 生产环境 APP_ENV ≠ development（否则 mock openid 让所有人共享一个账户）

第 1 项中风险、第 2-4 项都是高风险，建议上线前都修。

## 1. 数据隔离机制审计（你最关心的部分）

### 1.1 鉴权链路（OK ✅）

```
用户请求 → Authorization: Bearer <JWT>
           ↓
  get_current_user (deps.py)
    ├ 解码 JWT (HS256) → 拿 user_id (sub claim)
    └ session.get(User, user_id) → 从 DB 拿 User 对象
           ↓
  require_family
    └ 检查 user.family_id is not None
           ↓
  各 router (records / analytics / export / families)
    └ WHERE family_id = current_user.family_id
```

**关键设计正确点：**

- ✅ `family_id` 不从 JWT claim 取，而是每次请求都从 DB 查 User（`session.get(User, user_id)`）。这意味着：
  - 用户从家庭被移除/退出后，旧 token 立即失效（虽然 token 没显式 revoke，但 family_id 变 None 后 require_family 会拒绝）
  - 用户加入新家庭后，旧 token 也立即生效到新家庭
- ✅ JWT 解码失败/用户不存在/family_id None → 401/403 拦截
- ✅ token 有 30 天 TTL（虽然偏长，但不无限）

### 1.2 数据访问控制（OK ✅）

我把每个读取血糖数据的接口逐个审过：

| 接口 | 鉴权 | 是否过滤 family_id | 备注 |
|---|---|---|---|
| `GET /records` | require_family | ✅ `WHERE family_id = current_user.family_id` | OK |
| `GET /records/{id}` | require_family | ✅ `get_record_in_family` 检查 record.family_id == family.id，**不匹配返 404 而非 403**（防 ID 枚举）| OK |
| `POST /records` | require_family | ✅ 写入时 `family_id = current_user.family_id` | OK |
| `PATCH /records/{id}` | require_family | ✅ + 额外 recorder_id 检查 | OK |
| `DELETE /records/{id}` | require_family | ✅ + 额外 recorder_id 检查 | OK |
| `GET /analytics/matrix` | require_family | ✅ `get_records(..., family_id)` | OK |
| `GET /analytics/chart` | require_family | ✅ 同上 | OK |
| `GET /analytics/report` | require_family | ✅ 同上 | OK |
| `GET /families/me` | require_family | ✅ `session.get(Family, user.family_id)` | OK |
| `GET /export/csv` | 自定义 (token 或 header) | ✅ `stream_csv(..., family_id)` | **token 暴露问题，见 §2.1** |
| `POST /ai/parse-record` | require_family | N/A（不读 records，只解析文本） | 但有滥用风险，见 §2.7 |

**结论：** 没有任何接口绕过 `family_id` 过滤。**一个用户无法看到不属于自己家庭的记录。** ✅

### 1.3 防御性写法（OK ✅）

- ✅ ORM (SQLModel/SQLAlchemy) 参数化查询，防 SQL 注入
- ✅ get_record_in_family 返回 404 而非 403：避免攻击者通过状态码差异判断 record 是否存在
- ✅ Pydantic 校验 value 范围 `gt=0, lt=50`、note `max_length=200`
- ✅ DB 表加 CheckConstraint 兜底（value/period/source）
- ✅ JWT sub 强制转字符串，解码时容错（无效 → 401，不崩）

### 1.4 权限分级（OK ✅）

| 角色 | 能力 |
|---|---|
| 未登录 | `/auth/login` + `/health`（其他全部 401） |
| 登录但无家庭 | 只能看 `/health`、`/me`、调 `/families` 创建/加入（其他 403） |
| 家庭成员 (member) | 全部读、写自己记录、自己改/删自己记录、查看家庭信息 |
| 家庭管理员 (creator) | 上面所有 + 改家庭名/标准、移除成员、解散家庭、看邀请码 |

`require_creator` 链：require_family → role == 'creator'。OK。

**注意：** 当前管理员不能改家人的记录（前述 P0-DTL-1）。这个是 UX 问题，不是安全问题。

## 2. 漏洞/弱点（按风险等级）

### 🔴 S1（严重）：CSV 导出 query token 暴露

**位置：** `backend/app/routers/export.py:45-53`

```python
@router.get("/csv")
def export_csv(
    ...
    token: str | None = Query(default=None),
    authorization: str | None = Header(default=None),
    ...
):
    if token:
        current_user = get_user_from_query_token(token, session)
    else:
        current_user = get_current_user(authorization=authorization, session=session)
```

**问题：**
- 客户端用 `?token=<完整 30 天 JWT>` 形式 GET 导出（因为 wx.downloadFile 不能加 Authorization header）
- 这个 token 是**通用 30 天 JWT**，不是临时下载 token
- token 出现在：
  - 服务端 access log（nginx/uvicorn 默认记录 query string）
  - 客户端微信开发者工具 / 抓包工具
  - 如果用户分享导出链接，token 会被泄漏给收件人
- **后果：** 任何拿到这个 token 的人，30 天内可以**完全冒充用户**，包括读全部记录、新建/修改/删除、改家庭标准、移除成员、解散家庭

**修复（必做）：**

方案 A：签一次性短期 download token

```python
# security.py 加：
def create_download_token(user_id: int, family_id: int, ttl_seconds: int = 300) -> str:
    payload = {
        "sub": str(user_id),
        "scope": "export",
        "family_id": family_id,
        "exp": datetime.utcnow() + timedelta(seconds=ttl_seconds),
    }
    return jwt.encode(payload, settings.JWT_SECRET, algorithm=ALGORITHM)

# 加新接口：POST /api/v1/export/prepare → 返回短期 download_token + 完整 URL
@router.post("/prepare")
def prepare_export(
    from_: date = Query(alias="from"),
    to: date = Query(),
    current_user: User = Depends(require_family),
):
    download_token = create_download_token(current_user.id, current_user.family_id, 300)
    return {"download_url": f"/api/v1/export/csv?token={download_token}&from={from_}&to={to}"}

# /export/csv 改为只接受 scope='export' 的 token
def get_user_from_query_token(token, session):
    try:
        payload = decode_token(token)
        if payload.get("scope") != "export":  # 加这行
            api_error(401, "ERR_TOKEN_INVALID", "token scope 不正确")
        ...
```

方案 B（最小改动）：保持 query token 但限定 5min TTL

更简单但需要前端调整：先 POST `/export/prepare` 拿短期 token，再 GET `/export/csv?token=...`。

**优先级：上线前必修。**

### 🔴 S2（严重）：JWT_SECRET 默认空字符串

**位置：** `backend/app/config.py:14`

```python
JWT_SECRET: str = ""
```

**问题：**
- 如果生产环境的 `.env` 漏配 JWT_SECRET，启动后 secret 是空字符串
- 攻击者可以拿到任意 user_id（比如从客户端逆向、或者爆破自增 ID）→ 用空 secret 自己签 JWT → 完全冒充
- 启动时**没有任何检查**

**修复（必做）：**

`backend/app/config.py` 加：

```python
from pydantic import field_validator

class Settings(BaseSettings):
    ...
    @field_validator("JWT_SECRET")
    @classmethod
    def jwt_secret_required(cls, v: str, info) -> str:
        # 开发环境可以用默认 fallback；生产环境必须显式配
        if not v:
            env = (info.data.get("APP_ENV") or "").lower()
            if env == "production":
                raise ValueError("JWT_SECRET 必须在生产环境配置")
            return "dev-only-not-for-production-" + "x" * 32
        if len(v) < 32:
            raise ValueError("JWT_SECRET 长度必须 ≥ 32 字符")
        return v
```

**优先级：上线前必修。**

### 🔴 S3（严重）：邀请码缺 rate limit + 永久有效 + 不可重置

**位置：** `backend/app/routers/families.py:132` (`POST /families/join`)

**问题：**

1. **无 rate limit**：攻击者可以脚本化高速请求 `POST /families/join` 试码
   - 邀请码字符集 32 chars × 6 位 = 32^6 ≈ 10.7 亿组合
   - 假设攻击者每秒 100 个请求（保守）：单家庭命中需要 ≈ 124 天
   - 但如果有 1 万个活跃家庭，命中**任一**家庭只要 ≈ 17 分钟
   - 攻击成功 → 加入家庭 → 看到全部血糖记录
2. **永久有效**：`Family.invite_code` 没有 `expires_at`，从家庭创建那天起永久有效
3. **不可重置**：管理员无法重新生成邀请码（前端家庭管理页也没有这个功能）
4. **泄漏不可控**：管理员把邀请码截图发到群里、写进微信公众号，所有人永久能加入

**修复（建议在 v0.2 一并修，但 rate limit 上线前必加）：**

#### 3.1 加 rate limiting（必做）

用 SlowAPI 或自己写 Redis-based 限流：

```python
# pip install slowapi
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter

# families.py
@router.post("/join", response_model=FamilyDetailResponse)
@limiter.limit("10/minute")
def join_family(request: Request, req: JoinFamilyRequest, ...):
    ...
```

如果不想加 Redis，至少加内存级 IP+user 维度限流（10 次/小时）。

#### 3.2 加邀请码失败计数 + 临时锁定

```python
# 同一 IP 或 user 1 小时内连续 5 次"邀请码不存在"，拒绝该 IP 24 小时
```

#### 3.3 邀请码可重置 + 过期（建议 v0.2）

`Family` 表加：
```python
invite_code_expires_at: Optional[datetime] = None  # None 表示永久（保留兼容）
invite_code_rotated_at: datetime
```

家庭管理页加按钮"重新生成邀请码"。

#### 3.4 加入数限制

家庭最大成员数（比如 10 人），防被恶意加入大量假账号。

**优先级：rate limit 上线前必修；其他 v0.2。**

if settings.APP_ENV == "development" and settings.WX_MOCK_OPENID:

@field_validator("WX_MOCK_OPENID")

def mock_openid_only_in_dev(cls, v, info):

### 🟡 M2（中危）：measured_at 没有范围限制

**位置：** `backend/app/schemas/record.py:38-44`

```python
class CreateRecordRequest(BaseModel):
    value: float = Field(gt=0, lt=50)
    period: str
    measured_at: datetime  # 无任何限制
    note: Optional[str] = Field(default=None, max_length=200)
    source: str = Source.manual.value
```

**问题：**
- 攻击者（已加入某家庭的恶意成员）可以传 `measured_at = "2099-01-01"` 或 `"1900-01-01"`
- 数据库存入污染数据，分析报表显示异常
- 不会泄漏数据，但破坏数据完整性

**修复（建议）：**

```python
from pydantic import field_validator
from datetime import timedelta

class CreateRecordRequest(BaseModel):
    ...
    @field_validator("measured_at")
    @classmethod
    def measured_at_in_range(cls, v: datetime) -> datetime:
        now = datetime.utcnow()
        if v > now + timedelta(hours=1):
            raise ValueError("测量时间不能晚于当前时间")
        if v < now - timedelta(days=90):
            raise ValueError("测量时间不能早于 90 天前")
        return v
```

**优先级：v0.2，非 P0。**

### 🟡 M3（中危）：解散家庭后 GlucoseRecord 数据残留

**位置：** `backend/app/routers/families.py:210-223`

```python
@router.delete("/me", status_code=204)
def dissolve_family(...):
    family = get_my_family(...)
    members = session.exec(select(User).where(User.family_id == family.id)).all()
    for member in members:
        member.family_id = None
        member.role = None
        member.joined_at = None
        session.add(member)
    session.delete(family)
    session.commit()
```

**问题：**
- 删除 Family 行，但**不删 GlucoseRecord**
- 这些 record 的 family_id 指向已不存在的 family ID
- **当前实际：** 没人能再访问（因为没有 user.family_id 等于这个值），算"软隐藏"
- **未来风险：**
  - SQLite 主键 autoincrement 一般不复用，但理论可能（如果 sqlite_sequence 被重置）
  - GDPR / 用户数据删除权：用户解散后期望"完全删除"，但记录还在
  - 数据库膨胀

**修复：**

```python
@router.delete("/me", status_code=204)
def dissolve_family(...):
    family = get_my_family(...)
    
    # 先删所有血糖记录
    session.exec(
        delete(GlucoseRecord).where(GlucoseRecord.family_id == family.id)
    )
    
    # 重置成员 family_id
    members = session.exec(...).all()
    for member in members:
        member.family_id = None
        ...
    session.delete(family)
    session.commit()
```

或者更稳妥：用软删除（Family 表加 `deleted_at` 列），保留数据但所有查询过滤 `deleted_at IS NULL`。

**优先级：v0.2，非 P0。**

### 🟢 L1（低危）：JWT 一旦签发不可 revoke

**位置：** 整个 token 体系

**问题：**
- 30 天 JWT 一旦泄漏，30 天内有效
- 即使用户改密码、退出家庭，旧 token 还能访问（取决于具体接口的 require_family 检查，但很多场景下旧 token 仍然有用）

**修复（v0.2）：**

引入 server-side session 或 token blacklist：

- 方案 A：Redis 存活动 token（每次请求查一次）
- 方案 B：User 表加 `token_version`，签 token 时带 version，验证时对比；用户被移除/退出时 increment version → 旧 token 全部失效

**优先级：v0.2。**
